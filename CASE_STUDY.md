# Tower Defense Game — Описание кейса

## О проекте

**Tower Defense Game** — это полнофункциональная браузерная игра жанра Tower Defense, реализованная с использованием 3D-визуализации. Проект представляет собой монолитное приложение с разделением на клиентскую и серверную части, включающее редактор карт и игровой движок.

## Технологический стек

### Frontend
- **Vue.js 3** — реактивный UI-фреймворк
- **Babylon.js** — 3D-графический движок для рендеринга сцены
- **Vite** — современный инструмент сборки

### Backend
- **Fastify** — высокопроизводительный веб-фреймворк
- **Node.js** — серверная платформа

### Архитектура
- Монorepo-структура с использованием npm workspaces
- RESTful API для взаимодействия клиента и сервера
- Файловая система для хранения данных пользователей и карт

## Ключевые функции

### 1. Система авторизации
- Регистрация новых пользователей с генерацией API-ключей
- Аутентификация через заголовки запросов
- Изоляция данных между пользователями

### 2. Редактор карт
- Интерактивное размещение зданий и башен на сетке
- Визуальное планирование пути для врагов
- Поддержка различных размеров объектов (1x1, 2x2, 3x3)
- Режимы редактирования: размещение, удаление, редактирование пути
- Preview-режим с визуализацией валидности размещения

### 3. Игровой движок
- Система волн врагов с настраиваемыми параметрами
- Физика движения мобов по пути (Manhattan distance)
- Система боя: башни → снаряды → враги
- Динамическая система здоровья с визуальными индикаторами
- Экономика игры (золото, награды)
- Управление игровым временем (пауза, возобновление, изменение скорости)

### 4. 3D-визуализация
- Интерактивная камера с возможностью вращения и масштабирования
- Рендеринг сетки, зданий, пути, врагов и снарядов
- Оптимизированное управление памятью (dispose ненужных мешей)

## Проблемы и решения

### Проблема 1: Координация движения врагов по сетке

**Проблема**: Враги должны перемещаться по дискретной сетке, следуя ортогональному пути между waypoints, с корректной интерполяцией и учетом скорости.

**Решение**:
- Реализована система сегментного движения: враг проходит путь по сегментам между waypoints
- Использован параметр `t` (0-1) для интерполяции позиции внутри сегмента
- Расчет длительности сегмента на основе Manhattan distance и скорости врага
- Переход к следующему сегменту при достижении `t >= 1`

```javascript
// Из mobs.js - расчет движения по сегментам
const distCells = Math.abs(to.col - from.col) + Math.abs(to.row - from.row);
const segDuration = Math.max(0.0001, distCells / speed);
t += deltaSeconds / segDuration;
```

### Проблема 2: Валидация размещения объектов в редакторе

**Проблема**: Необходимо предотвратить размещение башен на пути врагов и наложение объектов друг на друга.

**Решение**:
- Реализована функция `canPlace()`, которая проверяет:
  - Границы сетки
  - Пересечение с ячейками пути (путь строится с помощью Manhattan distance между waypoints)
  - Коллизии с существующими объектами через `rectanglesOverlap()`
- Создан Set для быстрой проверки занятых ячеек пути
- Визуальная обратная связь через цвет preview-меша (зеленый/красный)

### Проблема 3: Система боя с отслеживанием целей

**Проблема**: Башни должны находить ближайшую цель в радиусе действия и выпускать снаряды, которые должны отслеживать движущуюся цель.

**Решение**:
- Реализована функция `findTarget()`, которая:
  - Перебирает всех врагов
  - Проверяет расстояние до цели (3D distance)
  - Выбирает ближайшего врага в радиусе атаки
- Снаряды используют динамическое отслеживание цели: на каждом кадре пересчитывают направление к текущей позиции цели
- Снаряды автоматически уничтожаются при достижении цели или при исчезновении цели

```javascript
// Из combat.js - динамическое отслеживание цели
const dir = to.subtract(proj.position);
dir.normalize();
proj.position = proj.position.add(dir.scale(speed * deltaSeconds));
```

### Проблема 4: Управление памятью в 3D-сцене

**Проблема**: При частом создании и удалении объектов (враги, снаряды) возникает утечка памяти в Babylon.js.

**Решение**:
- Реализован паттерн dispose для всех временных объектов:
  - Враги: dispose меша, HP-баров и материалов
  - Снаряды: dispose меша и материала при попадании или исчезновении цели
  - При перерисовке пути: удаление старых мешей перед созданием новых
- Использование lifecycle hooks Vue (`onBeforeUnmount`) для очистки при размонтировании компонента

### Проблема 5: Синхронизация игрового времени и рендеринга

**Проблема**: Игровая логика должна обновляться с учетом delta time, но также поддерживать паузу и изменение скорости.

**Решение**:
- Единый игровой цикл в `engine.runRenderLoop()`
- Расчет delta time с учетом множителя скорости (`speedMult`)
- Флаги состояния (`running`) для управления паузой
- Разделение логики обновления игрового состояния и рендеринга:
  - Обновление логики происходит только при `running.value === true`
  - Рендеринг происходит всегда для плавности интерфейса

### Проблема 6: Валидация и нормализация данных карты на сервере

**Проблема**: Клиент может отправить некорректные данные карты (отрицательные координаты, некорректные размеры, отсутствующие поля).

**Решение**:
- Использование JSON Schema для валидации входящих данных в Fastify
- Нормализация данных на сервере:
  - Ограничение значений через `Math.max` и `Math.min`
  - Установка значений по умолчанию для отсутствующих полей
  - Миграция старых форматов (legacy entries без size/height)
  - Валидация типов и диапазонов значений

```javascript
// Из map.js - нормализация данных
const normalized = {
  grid: {
    cols: Math.max(1, Math.min(1000, Number(incoming.grid.cols))),
    rows: Math.max(1, Math.min(1000, Number(incoming.grid.rows))),
    cellSize: Math.max(0.1, Math.min(100, Number(incoming.grid.cellSize)))
  },
  // ...
};
```

### Проблема 7: Конвертация между координатами сетки и мировыми координатами

**Проблема**: Необходимо конвертировать дискретные координаты сетки (col, row) в непрерывные 3D-координаты для размещения объектов.

**Решение**:
- Реализована функция `gridToWorld()`, которая:
  - Учитывает размер ячейки (`cellSize`)
  - Центрирует координаты сетки относительно начала координат
  - Возвращает мировые координаты для размещения мешей
- Обратная функция `pickGrid()` для определения ячейки по координатам курсора

```javascript
// Из createScene.js
export function gridToWorld(grid, col, row) {
  const x = (col + 0.5) * grid.cellSize - (grid.cols * grid.cellSize) / 2;
  const z = (row + 0.5) * grid.cellSize - (grid.rows * grid.cellSize) / 2;
  return new Vector3(x, 0, z);
}
```

## Архитектурные решения

### Модульная структура кода
- Разделение логики на модули: `createScene.js`, `mobs.js`, `combat.js`, `waves.js`
- Четкое разделение ответственности между компонентами Vue и игровой логикой
- Переиспользуемые функции для работы с сеткой и координатами

### API дизайн
- RESTful endpoints (`/auth/register`, `/auth/login`, `/map`)
- Аутентификация через заголовки для безопасности
- Изоляция данных пользователей на уровне хранилища

### Обработка ошибок
- Try-catch блоки в критических местах (инициализация, загрузка данных)
- Валидация ответов API с информативными сообщениями об ошибках
- Graceful degradation (использование дефолтных значений при ошибках загрузки)
